# Smart Fleet Fuel Optimization Pipeline (Phase 1 & 2)

A production-ready, serverless internet-of-things (IoT) telemetry ingestion pipeline built on Google Cloud Platform (GCP). The architecture processes real-time vehicle streaming diagnostics (RPM, speed, throttle position, fuel flow) generated by a distributed fleet simulation, securely capturing data streams with sub-second latency and auto-mapping properties directly into a time-partitioned BigQuery analytical data warehouse.

---

## 🏗️ System Architecture

```
+------------------------+      JSON Streams      +--------------------+      Direct Link      +--------------------------+
| Python Vehicle Fleet   | ---------------------> | Google Cloud       | --------------------> | BigQuery Data Warehouse  |
| Distributed Simulator  |   (Sub-second Push)    | Pub/Sub Ingestion  |   (Auto-Table Map)    | (Time-Partitioned Table) |
+------------------------+                        +--------------------+                       +--------------------------+

```

### Key Architectural Highlights

* **Zero-Compute Infrastructure (ELT Pattern):** Bypasses middleware compute engines (like Cloud Dataflow) for standard ingestion, utilizing a native GCP **Pub/Sub to BigQuery Subscription** link to minimize networking overhead and cloud spend.
* **Elastic Auto-Scaling:** Built on purely serverless architectural primitives that auto-scale to absorb high-throughput velocity spikes (supporting millions of messages/sec) without resource re-provisioning.
* **Storage Optimization:** Leverages BigQuery database time-partitioning to segment storage blocks by calendar date, maximizing execution speed and slashing analytics costs by limiting query data scans.

---

## 🛠️ Technology Stack & Dependencies

* **Cloud Platform:** Google Cloud Platform (GCP)
* **Ingestion Layer:** Google Cloud Pub/Sub
* **Data Warehouse:** Google Cloud BigQuery Studio
* **Simulation Runtime:** Python 3.12+
* **Transport Serialization:** Protocol Buffers (Protobuf) v3 & JSON Byte Streams

## 🚀 Step-by-Step Deployment Guide

Follow these sequential steps to establish the environment infrastructure, configure system identities, and activate the real-time data flow from a clean GCP account.

### Step 1: Clone Workspace & Set Local Variables

Open your **Google Cloud Shell Terminal** and configure the codebase environment.

```bash
# Clone the repository to your environment workspace
git clone https://github.com/YOUR_USERNAME/smart-fleet-fuel-optimization.git
cd ~/smart-fleet-fuel-optimization

# Explicitly bind your active Google Cloud Project ID to the shell terminal session
export GCP_PROJECT=$(gcloud config get-value project)

# Verify the project variable mapping (should output your unique project alphanumeric string)
echo $GCP_PROJECT

```

---

### Step 2: Establish Ingestion Pipeline Infrastructure

Execute the following native commands in your terminal to provision a clean Pub/Sub stream ingestion plane:

```bash
# Delete pre-existing testing topics to clear out strict schema locks
gcloud pubsub topics delete telemetry-raw --quiet 2>/dev/null || true

# Provision a clean enterprise-grade ingestion topic
gcloud pubsub topics create telemetry-raw

```

---

### Step 3: Setup the BigQuery Storage Layer

1. In the GCP Console, use the search bar or Navigation Menu to open **BigQuery Studio**.
2. In the left-hand **Explorer panel**, click the three vertical dots next to your project id (`dhana-gcp`) and select **Create dataset**.
* **Dataset ID:** `fleet_telemetry`
* **Data location:** `us-central1` (Keeps processing performance high and regional localized transport free)


3. Click the three vertical dots next to your newly created `fleet_telemetry` dataset folder and select **Create table**.
* **Table name:** `scored_events`
* Under **Schema**, toggle the **Edit as text** switch to active on the upper right side of the input module.
* Paste the following structurally optimized JSON array schema definition exactly:



```json
[
  {"name": "vehicle_id", "type": "STRING", "mode": "REQUIRED"},
  {"name": "timestamp", "type": "STRING", "mode": "REQUIRED"},
  {"name": "speed_kmh", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "rpm", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "throttle_pct", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "fuel_flow_lph", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "brake_event", "type": "BOOLEAN", "mode": "NULLABLE"},
  {"name": "brake_force_pct", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "braking_distance_m", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "acceleration_ms2", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "load_weight_kg", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "idle_duration_secs", "type": "FLOAT", "mode": "NULLABLE"},
  {"name": "road_grade_pct", "type": "FLOAT", "mode": "NULLABLE"}
]

```

4. Scroll down to **Partition and cluster settings**:
* **Partition by:** Select `timestamp`
* **Partitioning type:** Select `By day`


5. Click **Create table**.

---

### Step 4: Configure the Direct Link Identity Access Management (IAM)

Because we are using a direct architectural link, you must grant the internal platform Pub/Sub background engine systemic permissions to append database records into BigQuery.

1. Navigate to the **IAM & Admin ➔ IAM** page via the Cloud UI Console.
2. Check the box at the upper right that says **"Include Google-provided role grants"** (this unhides systemic infrastructure background accounts).
3. Search the list to locate your unique project's Pub/Sub service account identity. It will look exactly like this:
`service-[YOUR_GCP_PROJECT_NUMBER]@gcp-sa-pubsub.iam.gserviceaccount.com`
4. Click the **Edit Principal** (pencil icon) next to that row.
5. Click **➕ Add Another Role**, choose **BigQuery Data Editor** from the selection list, and hit **Save**.

---

### Step 5: Provision the Direct Streaming Link via Pub/Sub UI

1. Navigate back to the **Pub/Sub ➔ Topics** dashboard interface and click on your topic: `telemetry-raw`.
2. Click the **➕ CREATE SUBSCRIPTION** button at the top menu bar.
3. Configure the delivery link parameter configurations as follows:
* **Subscription ID:** `telemetry-bq-direct-sub`
* **Delivery type:** Select the radio button for **Write to BigQuery**
* **Select a BigQuery table:** Enter your exact target mapping signature: `dhana-gcp.fleet_telemetry.scored_events`
* **Schema properties:** Check the box that says **Use table schema**
* **Safety shield configuration:** Check the box that says **Drop unknown fields**


4. Leave all remaining configurations at defaults and click the blue **CREATE** button at the base of the page.

## 🚦 Executing the Processing Pipeline

With the cloud components interconnected, you can now instantiate the distributed physics simulator and audit the pipeline traffic.

### 1. Launch the Vehicle Simulator

Go back to your **Cloud Shell Terminal window** and spin up the multi-vehicle asset emulation loop:

```bash
cd ~/smart-fleet-fuel-optimization/simulator
python simulator.py

```

*The terminal will initialize and begin outputting transmission log hooks as it serializes and publishes vehicle data packs every 2 seconds.*

### 2. Audit Stream Logs and Metrics

To verify data delivery without querying the database, open the **Pub/Sub ➔ Subscriptions** dashboard and click on `telemetry-bq-direct-sub`:

* Under the **Metrics** tab, verify that your **Oldest unacked message age** line graph drops to flat zero. This indicates that BigQuery is accepting your data payload structures with zero resistance.

### 3. Run the Analytical Validation Scan

Open a blank SQL worksheet tab within **BigQuery Studio** and execute the following evaluation statement to view your streaming database entries live:

```sql
SELECT 
  vehicle_id, 
  timestamp, 
  speed_kmh, 
  fuel_flow_lph, 
  brake_event
FROM 
  `fleet_telemetry.scored_events`
ORDER BY 
  timestamp DESC 
LIMIT 10;

```
